# アレンジしたところ
元のコードは可読性、保守性に欠け、今回見たく 4x4,5x5 などにしたくなった場合に変更箇所がひと目でわからず、用意にバグをうみかねないコードだった
そのため、どんな数でも 1 行変えるだけで対応できるように、保守性の高いコードを書くようアレンジした。
また、
定数は UPPER_SNAKE_CASE
変数は lowerCamelCase
使ってほしくない変数やプライベートなものは _ スタートなど、その言語のしきたりに沿った書き方をした
また、できるかぎりconstを使用してグローバルスコープの汚染や、スコープ外からの値の変更などをできないようにした。
varなどのスコープ外からの値の変更やグローバルスコープの汚染などはバグを生み出しやすく、現在は推奨されていないので、できる限りモダンですっきりした書き方になおした。
# まなんだところ
誰にでもわかり、バグをうみにくく、保守しやすい安全なコードを書くことの大事さと難しさを学んだ。
自分しかわからないコードや、バグを生む危険性が少しでも高いコードを書くのは、属人性が生まれたりメンテのコストがかかるため良くない。
いきなりコードを書き始める前に、頭の中で良い設計をしてから、コードに落とし込んで行くことが大切だとおもった。
